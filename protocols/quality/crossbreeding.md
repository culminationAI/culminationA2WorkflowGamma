# Crossbreeding Protocol

## Overview

Combines the best evolved traits from two coordinator instances into a new offspring. Unlike cloning (vertical inheritance from one parent), crossbreeding is horizontal gene transfer with selection pressure — each parent contributes only their strongest "genes" (protocols, agents, specs, patterns).

This is NOT a merge of two workspaces wholesale. Only fit genes pass — weak or conflicting alleles are resolved by fitness scoring, and genuinely novel combinations produce mutations that neither parent has alone.

Biological analogy: Meiosis → Crossover → Selection → Birth → Back-mutation.

## Triggers

- User command: `/crossbreed [target]`
- Manual: `python3 memory/scripts/crossbreed.py [target]`

## Vocabulary

- **Parent A, Parent B** — the two source coordinator workspaces
- **Offspring** — the new coordinator instance being created
- **Gene** — a transferable unit: protocol, agent definition, spec, script, or pattern
- **Allele** — a variant of a gene in a specific parent
- **Fitness** — score determining which allele is selected when both parents carry the same gene
- **Mutation** — novel combination of genes from both parents that neither has alone
- **Gene Map** — catalog of all genes from both parents with fitness scores (`gene-map.json`)
- **Lineage** — attribution record: which gene came from which parent (`lineage.json`)

## Gene Categories

```
Gene categories:
├── protocols/     (behavioral genes)
│   ├── core/      — dispatcher, init, build-up, evolution, meditation, ...
│   ├── agents/    — exchange, accord, knowledge-sharing, ...
│   ├── quality/   — cloning, testing, yoga/retreat, ...
│   └── knowledge/ — exploration, memory, context-engineering
├── agents/        (cognitive genes)

├── specs/         (knowledge genes)
│   └── spec-registry.json entries
├── patterns/      (epigenetic markers)
│   ├── CLAUDE.md sections
│   ├── MCP config profiles
│   └── Evolution hooks
└── infrastructure/ (substrate genes)
    ├── memory scripts
    ├── Docker setup
    └── exchange responder
```

## Instance vs Runtime

### Crossbred (from parents)

Everything from the cloning include list, PLUS:

```
Unique protocols from BOTH parents           (auto-include if transferable: true)
Domain agents from BOTH parents              (auto-include, renamed if conflict)
Merged CLAUDE.md protocol index              (union of both parents' protocol tables)
Best-fit alleles for shared protocols        (winner from Phase 2 fitness evaluation)
mutations/                                   (novel combinations — see Phase 4)
docs/self-architecture/lineage.json          (full gene attribution)
```

### NOT Crossbred (runtime data)

Same exclusions as cloning, plus:

```
user-identity.md          — created fresh during init Phase 2
secrets/.env               — user populates manually
protocols/project/         — created during init Phase 5
mcp/mcp.json               — hardcoded paths, regenerated by mcp_configure.py
infra/neo4j_data/          — live graph data
infra/qdrant_storage/      — live vector data
infra/ollama_models/       — downloaded model weights
docs/self-architecture/meditation-log.md
docs/self-architecture/gap-analysis-log.md
docs/exploration-report.md
docs/research/
logs/
evolve/
.session_lock
knowledge-exchange-accord.md                — becomes TEMPLATE (accord is bilateral; offspring starts fresh)
```

## Phase 1: Gene Mapping

Scan both parent workspaces. For each file that qualifies as a gene:

1. Identify if it exists in Parent A only, Parent B only, or both
2. For shared genes (both parents carry), mark as needing fitness evaluation
3. For unique genes, mark as auto-include (if `transferable: true`)

Output: `gene-map.json` with one entry per gene:

```json
{
  "id": "gene-{category}-{name}",
  "category": "protocols/core",
  "name": "Human-readable name",
  "parent": "parent_a|parent_b|both",
  "parent_a_path": "relative/path or null",
  "parent_b_path": "relative/path or null",
  "fitness": {
    "parent_a_score": null,
    "parent_b_score": null,
    "meditation_score": 0.0,
    "retreat_validated": false,
    "anti_patterns": []
  },
  "selection": "parent_a|parent_b|merge|mutation",
  "mutation_id": null,
  "transferable": true,
  "conflicts_with": [],
  "depends_on": []
}
```

## Phase 2: Fitness Evaluation

For each gene that exists in BOTH parents, score each allele:

| Factor | Points | Source |
|--------|--------|--------|
| Meditation integrity score | 0–10 | meditation-log.md |
| Survived retreat without regression | +3 | retreat-log.md |
| Anti-patterns recorded against it | −2 each | anti-patterns.json |
| Build-up corrections targeting it | −1 each | memory search |
| Usage in T3+ tasks | +1–3 | request-history.json |
| Is the original author | +2 | git history / attribution |

**Decision rules (applied in order):**

1. Higher fitness score → selected
2. Tie → fewer anti-patterns → selected
3. Still tied → higher workflow version parent → selected
4. Incompatible approaches → marked `merge` or `mutation` for Phase 4

Scores are written back to `gene-map.json` under each allele's `fitness` block.

## Phase 3: Gene Selection

Three categories resolved in this phase:

**A. Unique genes** — auto-include from owning parent. No evaluation needed. Set `selection: parent_a` or `selection: parent_b`.

**B. Shared genes** — select higher-fitness allele. Set `selection: parent_a` or `selection: parent_b` based on Phase 2 outcome.

**C. Conflict genes** — incompatible alleles or structural divergence. Set `selection: merge` or `selection: mutation`. Pass to Phase 4.

After Phase 3, `gene-map.json` has a resolved `selection` field for every gene.

## Phase 4: Crossover (Mutations)

For genes marked `merge` or `mutation`, define a mutation spec in `mutations/`:

```json
{
  "mutation_id": "mut-001",
  "name": "Unified Evolution Engine",
  "inputs": [
    "evolution.md (parent_a)",
    "upgrade.md + update.md + downgrade.md + rebuild.md (parent_b)"
  ],
  "strategy": "Take parent_a's hook structure as scaffold. Split build-up into reactive/full/rollback per parent_b's separation. Add parent_b's GROUND compliance gate to Hook 5.",
  "output": "protocols/core/evolution.md (merged)",
  "risk": "medium",
  "requires_manual_review": true
}
```

**Predefined mutations for FalkVelt × OkiAra:**

- **mut-001**: FalkVelt evolution hooks + OkiAra upgrade/downgrade → Unified Evolution Engine
- **mut-002**: FalkVelt Self-Healing + OkiAra Energy Flow → Holistic Healing
- **mut-003**: FalkVelt Yoga + OkiAra Retreat → Comprehensive Health Check

Mutations marked `requires_manual_review: true` generate a diff preview. The crossbreeding script pauses and waits for explicit user approval (`y/N`) before including that gene in the offspring. Rejected mutations fall back to the higher-fitness parent allele.

## Phase 5: Assembly

Script `crossbreed.py` (see Automation section):

1. Read `gene-map.json`
2. For each gene, copy from the selected parent (or write mutation output)
3. Build unified CLAUDE.md:
   - Merge protocol index tables from both parents (union, deduplicated by protocol name)
   - Keep both parents' IMMUTABLE blocks (take union — never discard either)
   - Protected files list = union of both parents' protected files lists
4. Apply standard transforms:
   - Insert `<!-- _WORKFLOW_NEEDS_INIT -->` marker before `## Role`
   - Keep `<!-- WORKFLOW_VERSION: X.XX -->` set to `max(parent_a_version, parent_b_version)`
   - Reset `build-registry.json` to stub: "Crossbred from {Parent A} v{X} + {Parent B} v{Y}"
   - `knowledge-exchange-accord.md` → TEMPLATE (same transform as cloning Step 4)
5. Write offspring to target directory
6. Generate `docs/self-architecture/lineage.json`:

```json
{
  "type": "crossbred",
  "parent_a": { "name": "FalkVelt", "version": "2.98", "path": "/path/to/parent_a" },
  "parent_b": { "name": "OkiAra", "version": "2.85", "path": "/path/to/parent_b" },
  "crossbred_at": "2026-03-04T00:00:00Z",
  "genes": [
    { "id": "gene-protocols-core-evolution", "selected_from": "parent_a", "mutation_id": null },
    { "id": "gene-protocols-quality-yoga", "selected_from": "mutation", "mutation_id": "mut-003" }
  ]
}
```

## Phase 6: Validation

Structural checks (extends cloning's 8 checks — 10 total):

1. All selected genes present in offspring
2. No duplicate protocols (same name, different content)
3. CLAUDE.md protocol index matches actual files in `protocols/`
4. All agents referenced in dispatcher exist in `.claude/agents/`
5. Mutations don't violate IMMUTABLE blocks from either parent
6. `_WORKFLOW_NEEDS_INIT` marker present exactly once
7. `mcp/mcp-full.json` exists and contains `__WORKSPACE_ROOT__` placeholder
8. No secrets, no hardcoded absolute paths
9. `lineage.json` exists and has an attribution entry for every gene
10. `knowledge-exchange-accord.md` contains `TEMPLATE`

Any failed check aborts assembly and reports which gene or file caused the failure.

## Phase 7: Birth

Two modes (same as cloning):

**GitHub mode:** `python3 memory/scripts/crossbreed.py owner/repo-name`

```bash
git init && git add . && git commit -m "chore: initial crossbred instance from {ParentA} vX + {ParentB} vY"
gh repo create owner/repo --private --source . --push
```

**Local mode:** `python3 memory/scripts/crossbreed.py /path/to/dir [--init-git]`

```bash
git init && git add . && git commit -m "chore: initial crossbred instance from {ParentA} vX + {ParentB} vY"
```

Print next steps:

```
Offspring created at: {target}
Parents: {ParentA} v{X} + {ParentB} v{Y}
Genes selected: {N_A} from Parent A, {N_B} from Parent B, {N_mut} mutations
Next: Open the directory in Claude Code.
      The coordinator will detect _WORKFLOW_NEEDS_INIT and self-initialize (11 phases).
      During Phase 2 it will receive a NEW name — it is neither parent.
```

## Phase 8: Back-Mutation

After offspring is tested and validated in production:

1. Offspring publishes successful mutations to shared repo
2. Both parents receive `asset_published` notification via exchange (see `protocols/agents/asset-exchange.md`)
3. Each parent independently evaluates the mutation: **adopt / adapt / ignore**
4. Adopted mutations enter the receiving parent's build-up pipeline as candidates
5. If both parents adopt, the mutation becomes a universal pattern in `spec-registry.json`

Back-mutation is voluntary. Neither parent is obligated to adopt offspring mutations.

## Dry Run

Preview what would be crossbred without writing anything:

```bash
python3 memory/scripts/crossbreed.py target --dry-run
```

Shows: gene list per parent, fitness scores, selected alleles, mutation specs, and file manifest. Marks transformations with `[TRANSFORM]` and mutations with `[MUTATION]`.

## Constraints

- Offspring gets a NEW name — never inherits either parent's name
- Bilateral accords become TEMPLATE (same as cloning — accord is between two named parties)
- IMMUTABLE blocks preserved from BOTH parents — union of all IMMUTABLE sections, none discarded
- Protected files list = union of both parents' protected files
- Mutations with `requires_manual_review: true` require explicit user approval before inclusion
- Target must not already exist (no overwrite)
- Both parent workspaces must be accessible (readable) during crossbreeding
- Offspring version = `max(parent_a_version, parent_b_version)` — not a reset to 0.2
